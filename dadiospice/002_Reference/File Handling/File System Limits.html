File System Limits 文件系统限制

在这里，你可以找到信息文件系统是如何工作的和它的限制。

当使用文件系统的函数包括 GameMaker:Studio ，这是非常重要确切地知道一切正常，什么事情会在幕后。为此，本节的目的是解释和澄清到底事情是如何存储的，它们存储的地方，这个系统有什么可能的限制或变通。

沙盒

（最重要的）的事情要注意有关该文件的功能是，他们是有限（一般）的沙箱。这句话的意思是，GameMaker:Studio 无法保存或加载文件从任何地方，是没有的游戏捆绑或本地存储设备的一部分没有明确的从用户输入，和即使在当时这种导入是有限的只有Windows和Mac目标平台。

要了解沙箱首先你需要了解的文件，有两个不同的领域：

  1. The File Bundle - 文件捆绑 - 这是所有的游戏可执行文件，这些文件存储，包括您已经添加的文件，包括文件资源列表从GameMaker:StudioIDE中（这是外部文件要打包到本游戏）。
  2. The Save Area -保存区域 - 这是可以被安全地写入到由游戏和游戏，这是一个可写区域，保证设备存储的区域。

使用文件函数

这两个目标区（套装目录和本地存储区域）每个目标平台上可用，但在每一个他们的工作方式略有不同。但是 GameMaker:Studio已经抽象出来什么可以和什么不能做的主要精髓，使它更容易重新目标游戏多个环境。

首先，你应该明白是什么意思当我们谈论 working_directory。这是两种可能的保存位置（如上图中所示）的代名词，当您使用该目录中，它会做两件事情，这取决于您是否读取或写入，而你正在读取的文件是否已经改变与否

  1. Reading Operations - 读取操作 - working_directory会先检查，看是否有被访问的文件是保存区域，如果它是被使用。但是，如果该文件不存在保存区域，然后检查文件捆绑面积和用途（如果该文件被找你的游​​戏的包含文件）。
  2. Writing Operations - 写入操作 - 这只能发生在保存区。

使用这两个简单的规则，我们现在可以看到以下函数是如何工作的（这些例子来帮助你的过程可视化提供不同的函数）：

  • directory_create （和其他目录的功能）将只保存工作。
  • file system 函数会返回束和保存区域文件的信息，但只会写保存区域。
  • 写入文本，二进制或ini文件的功能，将创建一个新的文件保存，如果一个已经不存在，复制与游戏捆绑他们应该存在的原始文件的任何信息。

在沙箱外的保存和载入

在Windows平台上，有一个方法保存和加载文件的沙盘外，那就是使用的函数 get_open_filename 和 get_save_filename。这些函数都要求用户选择一个区域，用于加载和保存，并返回的字符串可以被用来在其余的文件功能，绕过沙箱。

特定平台的保存/载入

当GameMaker:Studio工作时，你应该知道，教目标平台都有自己的文件和目录可以写入和读取保存区域。下面是这些领域对每个目标的列表：

  • Windows - Windows的所有文件在 %localappdata%\<Game Name> 文件夹 (Windows 7 是在 /Users/<User Name>/AppData/Local/<Game Name> 文件夹).

  • Mac OS - 存储将取决于应用程序是否是沙箱或不是（苹果规则）

  • Ubuntu (Linux) - 文件存储在 Home/.config/gamename ， "Home" 用户家庭文件夹 - /home/<username>
  • HTML5 / Windows 8 (JS) - 一切都是通过本地存储

  • iOS -  存储是标准的位置（如观看通过iTunes）。

  • Android - 文件是在标准位置（这是不可见的，除非该设备是rooted的） /data/data/<package name>.

  • Windows Phone - 档案是在相同的位置的应用程序。

虽然在一些平台上你可以使用的文件名，它自己访问包含文件，这是值得注意的，在设备（android，iOS，Windows phone）你必须使用访问存储文件working_directory的前缀，如使用正好相对路径将无法正常工作。还值得指出的是HTML5和 Windows 8的目标模块在本地存储上有限制（可以是介于1MB和5MB取决于浏览器），这意味着你将不得以节省大量的精灵，截图等......
