Using Buffers 使用缓冲区

在这里，你可以得到一个概述缓冲区是如何工作的，他们在游戏中如何使用。

GameMaker:Studio已经推出了一系列游戏制作语言（GML）函数处理缓冲区。因为它是用来处理计算机和编程所有的时候，大多数人应该熟悉这个术语，但知道这个词其实并不意味着你知道这意味着什么。所以这个页面的目的是解释什么缓冲，以及如何使用它们在GameMaker:Studio编程的范围内，虽然他们的工作方式是相同的，无论语言或技术，这就是为什么他们的原因之一如此重要。

什么是一个缓冲区

缓冲区（编程）基本上是系统内的存储器，用于存储（例如数据传输，碰撞，彩色数据等..）的小数据包的数据的空间。既然是在系统内存中举行，这是非 常快的访问，一个缓冲区通常会被用于非常短期存储，就像在处理之前，接收网络信息，或存储在你的游戏中的一个检查点（这是在进一步解释例子在后面页）。 创建缓冲区，在系统内存中分配空间，以字节为单位计算，然后只要保留你的游戏你的游戏正在运行或使用相应的函数，直到你删除缓冲区。这意味着，即使你的游 戏是不是焦点（例如，在移动设备上，当你把游戏一定会被放到后台）的缓冲区仍然会存在，但如果游戏被关闭或重新启动一个呼叫缓冲区将被丢失。

NOTE: 重新启动游戏将无法清除或删除缓冲区！但它会防止任何进一步的id手柄将已经失去了先前创建的缓冲区，导致内存泄漏你的游戏最终会崩溃。所以，当重新开始游戏，记得先删除缓冲区。

缓冲区类型

GameMaker:Studio允许四种不同的缓冲区类型的创建。这样做的原因是，缓冲区被设计成一个高度优化的临时存储介质，因此你应 该创建一个缓冲区，是合适的，你希望它来存储数据的类型，否则你可能会得到错误或造成瓶颈在你的代码中。在进一步解释之前，让我们来看看四个可用的缓冲类 型（定义在GML常量）：

Constantdescription
buffer_fixed
一个固定大小的缓冲区，以字节为单位。的大小被设置创建缓冲区时，不能再次更改。

buffer_grow
缓冲区将增长将数据添加动态。创建它的初始大小（这应该是一个近似的预期要存储的数据的大小），然后将它扩展接受进一步的数据，这溢出的初始大小。

buffer_wrap
缓冲区数据将包装。当所添加的数据的缓冲区的大小达到限制值时，覆盖将被放回到上面开始的缓冲区，并进一步写入将继续从该点。

buffer_fast
这是一个特殊的“stripped down”极快的读/写缓冲区。然而，它只能被用于有buffer_u8和buffer_s8的数据类型，和必须为1字节对齐。 （数据类型和字节对齐的信息可以发现本页下方）。


这些都是缓冲区类型提供给您GameMaker:Studio，你选择哪一个将在很大程度上取决于你希望把它的使用。例如，一 个成长缓冲将被用于存储数据的“snapshot"”来创建一个保存游戏，因为你不知道将被放置在数据的实际总额，或快速缓冲时将使用你我知道，您正在使 用的值介于0和255或-128和127之间，例如，当RGB图像数据处理。

创建一个缓冲区时，您应该总是​​尝试创建一个大小是合适的类型，与一般的规则是，它应该创建适应，它是存储数据的最大大小，如有疑问，使用grow缓冲区，以防止改写错误。

实际的代码来创建一个缓冲区会是这个样子：

player_buffer = buffer_create(16384, buffer_fixed, 2);

这将创建一个固定的的的共16348个字节和字节对齐到2的缓冲区，则函数返回一个独特的id值，id值是存储在一个变量中，是为后面引用这个缓冲区。现在我们有一个缓冲区基本解释了，你应该学有关数据类型和字节对齐的的基本概念。

数据类型

当读取和写入数据的缓冲区，通过''data type''你用它在“chunks”定义数据。 ''data type'''设置分配的缓冲区内的价值被写入的字节数，它是必不可少的，你要正确获取，否则你会为您的代码得到一些非常奇怪的结果（甚至是错误的）。

缓冲区被写入（读出）顺序，陆续在那一个被写入的数据块，每一块是一组类型的数据。这意味着，理想情况下，你应该知道你是什么样的数据在任何时候写入到缓冲区。这些数据类型定义在GML由以下常量：

Data Type ConstantBytesDescription
buffer_u81
一个无符号8位整数。这是一个正的值从0到255。

buffer_s81
有符号8位整数。这可以是一个正或负的值，从-128到127（0个被归类为正性）。

buffer_u162
一个无符号16位整数。这是一个正的值从0 - 65,535。

buffer_s162
有符号16位整数。这可以是一个正或负的价值从-32,768到32,767（0被归类为正性）。

buffer_u324
一个无符号32位整数。这是一个正的值从0到4294967295。

buffer_s324
有符号32位整数。这可以是一个正或负的价值从-2,147,483,648到2,147,483,647（0被归类为正性）。

buffer_f162
一个16位浮点数字。作为16位有符号整数，这可以是一个正或负的值相同的范围内。 （目前不支持！）

buffer_f324
一个32位浮点数字。这可以是一个正或负的值相同的范围内，作为一个32位有符号整数。

buffer_f648
64位浮点数字。这可以是正值或负值 -(263) to 263 - 1.

buffer_bool1
一个布尔值。只能是1或0（true或false）

buffer_stringN/A
这是一个UTF-8空值终止字符串（0X00）。基本上GameMaker字符串被扔在缓冲区中，并在最后放一个0。


所以，说你已经创建了一个缓冲，你想要写的信息，那么你可以使用类似下面的代码：

buffer_write(buff, buffer_bool, global.Sound);
buffer_write(buff, buffer_bool, global.Music);
buffer_write(buff, buffer_s16, obj_Player.x);
buffer_write(buff, buffer_s16, obj_Player.y);
buffer_write(buff, buffer_string, global.Player_Name);

在上面的例子中，你可以看到，你可以写不同类型的数据的缓冲区（只限于一个特定的数据类型时，使用快速缓冲类型），而这一数据将连续被添加到缓冲区（尽管其缓冲区中的实际位置，将取决于它的字节对齐，解释见下文）。这是相同的，从缓冲区中读出信息，并在上面给出的例子的情况下，你会在写数据相同的顺序，从缓冲器读出，检查相同的数据类型，例如：

global.Sound = buffer_read(buff, buffer_bool);
global.Music = buffer_read(buff, buffer_bool);
obj_Player.x = buffer_read(buff, buffer_s16);
obj_Player.y = buffer_read(buff, buffer_s16);
global.Player_Name = buffer_read(buff, buffer_string);

正如你可以看到，你读出信息以相同的顺序，你将它读入缓冲区。如何添加和删除数据从缓冲区欲了解更多信息，请参阅下面的示例部分。

字节对齐

如果你已经读通过这个页面，你会看到一个缓冲区的字节对齐的引用。这基本上是指，新的数据将存储在一个给定的缓冲区内的位置。这是如何工作？那么，对于一 个单一的字节对齐的缓冲去中，每一块的数据被写入到缓冲区，按顺序，直接添加后，以前的每个新的数据枚。然而，2个字节对齐的缓冲，将每一块间隔时间为2 个字节的数据写入，所以，即使你的初始写入1个字节的数据，下一次写将移动两个字节对齐。 所以，如果你的字节对齐方式设置，比方说，4个字节，你写了一个单件的大小为1个字节的数据，然后做一个缓冲区（读/写缓冲区）获取当前的位置，会得到一个偏移量为1（在这种情况下的偏移量的字节数从缓冲区的开始到当前读/写位置）。

不过，如果你写的另一块数据，还有1个字节的大小，然后做一个缓冲告诉，你会得到一个偏移5个字节对齐（即使你只写了2个字节的数据）填充数据将其与4字节的缓冲区对齐。

基本上，这是什么意思是对准只会影响那里的东西是写，因此，如果你做一个缓冲告诉你写的东西后，它会返回当前写入的位置，紧跟着你以前写的数据。但是，请注意，如果你写的另一块数据，内部缓冲区写入位置移动沿对齐大小的下一个倍数的实际写入的数据块之前。
