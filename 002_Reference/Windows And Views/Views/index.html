Views 视野

本章节设计定义视野属性的函数。

This section deals with the functions for defining view properties.

在你可以操作GameMaker:Studio功能中，视野是一个最重要的显示属性。基本上，它们是游戏世界中的小窗口，向玩家显示房间的一部分。当你的游戏房间比能显示的画面更大时，视野会缩放或者以1：1的大小显示。关于如何不用代码直接在房间编辑器中设置视野，请了解这个章节 More About Rooms - Views.

GameMaker:Studio允许8个独立的视野（编号0 - 7），所有这些都可以在任何一个时间被激活，且都可以显示同一房间内的不同的部分。这意味着您可以使用视野来绘制HUD元素或屏幕分割效果。每个视野也有它自己的关联的视野输出端口。如果视野在房间里是可视的，则将绘制输出的部分。如果视野取景的大小与视野的输出不同，视野自身将被扭曲或缩放。 

关于视野和视野输出需要注意的是，游戏第一个房间的所有激活的视野输出端口，它们的边界框的整个区域便可以定义背景画布（或是MacOS和 Windows的窗口大小）的大小，请见下图说明：

Views are one of the most important of the display properties that you can manipulate with GameMaker:Studio. They are, basically, little windows into your game world that enable you to show the player parts of a room, either scaled or 1:1, and as such they are essential when your game room is larger than the display size. For information on how views can be set up without code directly from the room editor, please see the section More About Rooms - Views. 

GameMaker:Studio permits you 8 independent views (numbered from 0 - 7), all of which can be active at any one time, and all of which can show different parts of the same room. This means that you can use views to draw HUD elements or to have split screen effects for example. Each view also has it's own associated view port. If the view is a section of the room that the player can see, the port is the area of the screen on which this view is drawn, and as such you can distort and scale the view if it is a size other than the view port. 

Something to note about views and view ports is that the total area of the bounding box for all active view ports in the first room of the game is what defines the background canvas size (or window size for MacOS, Ubuntu (Linux) and Windows), as illustrated by the image below:

 

最后，你应该小心使用多个视野，因为所有实例的绘制事件每个视野都会调用一次，如果一个房间激活了三个视野，绘制事件将每步运行一次（运行了三次），这将降低大型或复杂游戏的运行速度。不过可以使用 view_current 变量限制这些绘制的响应。

Finally, you should take care when using multiple views as the draw event for all instances is called once for each view, so if you have three views active in one room, the draw event will be run three times every step (basically doing three times the work) which can be a cause for slowdown if the game is large or complex. The view_current variable can be used to help limit these draw calls however.
